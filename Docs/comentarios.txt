Instalação Padrão do playwright:
npm init playwright@latest

Nosso projeto utilizará as seguintes abordagens:
- Separação dos cenários de negócio dos scripts de automação
  - vamos precisar de no mínimo 2 arquivos
  
  ---> pasta features que contem arquivos:
  .feature - Funcionalidades e Cenários de negócio / Cristériois de Aceitação / Dados
  ---> pasta steps que contem arquivos:
  spec.js - Script de automação de teste

      Instalações adicionais:
      // Instala o pacote que dá suporte ao BDD
      npm install -D playwright-bdd
      npx bddgen init

      Feature: Login
       Scenario: Login with sucess
          Given
          When
          Then

    Funcionalidade: Login
          Cenário: Login com Sucesso
          Dado
          Quando
          Entao

Nomenclatura
- BDD - Behavior Driven Development - Desenvolvimento Direcionado pelo Comportamento
Abordagem de desenvolvimento, muito valorizada dentro da Agilidade
Objetivo é facilitar a comunicação entre os membros do time, áreas e clientes

- Gherkin: as instruções do BDD são escritas na sintaxe (formato) chamado Gherkin

- Cucumber/Behave/SpecFlow são exemplos de plugins que fazerm a integração entre arquivos .features com arquivos spec.js

- Page Objects / Page Factory: separar o mapeamento dos elementos da interface gráfica (páginas, telas) do código da automação
---> Pages com mapeamento de elementos
---> Steps com os scripts de automação
---> Features com os cenários de teste

Em um Projeto Simples cada teste é um arquivo é um arquivo .spec.js diferente
Em um projeto com Gherkin, cada teste é composto por um arquivo .feature + um arquivo steps.js

Diferenças de sintaxe conforme o type definido no package.json:

quando coloca commonjs ---> ele quer que use const ... = require('...')
quando coloca module   ---> ele quer que use import ... from ...


Sobre execução sincrona (uma parte espera a outra / Ex. do telefone) e
assincrona (a parte emissora não espera / Ex. da mensagem do whatsapp)

Sem async/await: transação sincrona
Com async/await: transação assincrona

Estamos usando dois frameworks: cada um deles tem um relógio de tempo limite para executar 
- playwright --> quem vai controlar o browser --> timeout --> msg mais específica que pode explicar a causa
- cucumberJS --> ele ajuda a organizar, é um plugin de organização --> timeout (msg genérica e não explicar a causa - não )
